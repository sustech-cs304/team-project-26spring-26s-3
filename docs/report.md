# Part I. Architecture & UI Design

## 一、Architecture Design

### 1.1 架构概述

本项目是一款面向 HarmonyOS 平板设备的数字笔记本应用，核心目标是为用户提供笔记本管理、页面组织和手写绘图等功能。结合当前系统实现与整体模块划分情况，本项目采用的是**分层架构（Layered Architecture）**，并在表示层使用了**MVVM 风格（MVVM-style Presentation Layer）** 的设计方式，同时结合 **Repository 模式** 完成业务逻辑与本地数据持久化之间的解耦。

这一架构并不是严格意义上的 MVC。原因在于，系统页面并不是主要通过统一的 Controller 来承担界面协调职责，而是通过 ViewModel 管理界面状态、协调业务操作、调用用例与仓储接口，并最终驱动界面更新。因此，将本项目描述为“分层架构 + MVVM 风格表示层 + Repository 数据访问模式”更加准确，也更符合当前代码结构。

本报告配套提供了一张高层架构图，用于展示系统的主要分层结构、关键模块以及模块之间的调用关系。

### 1.2 总体架构说明

从整体上看，系统可以划分为以下几个主要层次：

- 应用启动与导航层
- 表示层
- ViewModel / 表现逻辑层
- 领域层
- 数据层

其中，上层主要负责界面展示与交互，下层主要负责业务抽象与本地数据持久化，各层之间职责边界比较明确。这种设计有助于降低模块耦合度，使系统在开发、测试和后期维护时都更加清晰。

### 1.3 各层职责说明

#### （1）应用启动与导航层

应用启动后，由 `EntryAbility` 作为系统入口加载初始页面；`Index` 页面负责应用的初始化与重定向；`AppInitializer` 完成应用启动阶段的基础准备；`AppRouter` 负责页面跳转与路由组织，并将用户导向 Home、Notebook List 和 Editor 等核心界面。

这一层的职责相对集中，主要解决“应用如何启动”和“页面如何切换”这两个问题。通过将启动逻辑和路由逻辑单独抽出，系统的页面结构更加清晰，也便于后续增加新的页面模块。

#### （2）表示层（Presentation Layer）

表示层由 ArkUI 页面与组件构成，是直接面向用户的部分。当前系统的表示层主要包含以下三个功能模块：

- **Home Feature**：负责展示首页和快速操作入口；
- **Notebook Feature**：负责笔记本列表展示，以及新建、排序、重命名、删除等管理操作；
- **Editor Feature**：负责页面编辑、工具切换和画布书写。

这一层的主要任务是负责界面布局、视觉展示和用户交互事件采集，而不是直接处理底层数据存储或复杂业务逻辑。也就是说，表示层更关注“用户看到了什么”和“用户做了什么”。

#### （3）ViewModel / 表现逻辑层

在表示层之下，系统通过 ViewModel 连接界面和业务逻辑。当前主要包含以下三个核心 ViewModel：

- `NotebookListViewModel`：负责加载笔记本列表、创建笔记本、重命名、删除以及切换排序方式；
- `EditorViewModel`：负责加载笔记本信息、页面列表，以及页面的创建、删除、重排和画布元数据读取；
- `DrawingEditorViewModel`：负责管理绘图状态、当前工具参数、保存笔迹、撤销、重做和清空等编辑操作。

ViewModel 层是本项目中非常关键的一层。它使页面不再直接依赖底层仓储实现，也避免了大量业务逻辑直接堆积在页面文件中。这样不仅使代码结构更加清晰，也有利于团队成员并行协作。

其中，`DrawingEditorViewModel` 是编辑器模块的核心状态管理单元。它与 `CanvasBoard`、`ToolPropertyPanel` 等组件协同工作，使编辑页能够在保持较高交互复杂度的同时，仍然维持较好的结构可读性。

#### （4）领域层（Domain Layer）

领域层负责抽象系统中的核心业务对象和业务操作规则，主要包含以下三类内容：

- **实体（Entities）**：例如 `Notebook`、`NotebookPage`、`NotebookPageCanvas`、`Stroke` 和 `ToolSetting`；
- **仓储接口（Repository Interfaces）**：例如 `NotebookRepository` 和 `EditorRepository`；
- **用例（Use Cases）**：例如创建笔记本、获取页面列表、创建页面、保存笔迹、撤销和重做等操作。

这一层的作用在于把“系统的核心业务逻辑”从“具体实现方式”中分离出来。通过实体、接口和用例的抽象，系统上层无需关心数据最终是存储在文件、Preferences 还是其他存储介质中，这为项目后续扩展提供了良好的基础。

#### （5）数据层（Data Layer）

数据层负责本地数据的读写与持久化实现。当前系统采用本地优先（Local-first）的设计，不依赖后端服务器。主要组件包括：

- `NotebookRepositoryImpl`：负责笔记本、页面及画布相关数据的本地存取；
- `EditorRepositoryImpl`：负责单页笔迹数据的读写与持久化；
- `PreferencesDataSource`：负责排序方式等轻量级配置数据的保存；
- `FileDataSource`：负责笔记本、页面、画布等结构化数据的本地文件读写。

这种设计方式比较适合当前课程项目。一方面，它减少了服务端依赖，降低了实现复杂度；另一方面，它也能够保证用户笔记数据可以在设备本地稳定保存，满足离线使用场景。

### 1.4 模块之间的交互方式

系统的基本交互流程如下：

1. 用户启动应用后，由入口模块完成初始化并进入主界面；
2. 用户在页面中进行操作，例如查看笔记本、创建页面或开始书写；
3. 页面将交互事件传递给对应的 ViewModel；
4. ViewModel 调用领域层中的 Use Case 或 Repository Interface；
5. 仓储实现类将操作转化为本地文件或 Preferences 的读写；
6. 数据返回后，ViewModel 更新状态，页面根据状态变化刷新界面。

对于编辑器模块，其交互流程更加典型。`CanvasBoard` 负责接收手写笔或触控输入，`DrawingEditorViewModel` 负责维护当前绘图状态，`StrokeController` 用于生成和规范化笔迹数据，`UndoRedoController` 管理撤销与重做历史，`StrokeRenderer` 负责将笔迹绘制到画布上，最终由 `EditorRepositoryImpl` 将完成后的笔迹保存到本地。

这意味着编辑器内部形成了一个较为完整的闭环，即“输入事件采集 - 状态更新 - 业务处理 - 数据持久化 - 界面回显”。这种模块分离方式有利于保证编辑器的可维护性，也更适合高频交互的手写场景。

### 1.5 为什么这种架构适合本项目

本项目采用上述架构，主要有以下几个原因。

首先，分层设计使系统结构更加清晰。页面、状态管理、业务抽象和数据持久化分别由不同层承担，降低了模块间的耦合度，使系统更容易理解和维护。

其次，MVVM 风格使界面代码保持简洁。页面主要负责布局和事件绑定，ViewModel 负责状态与逻辑协调，这种方式特别适合当前项目这样具有多个主界面和较复杂编辑交互的应用。

再次，Repository 模式提高了系统的扩展性。虽然当前系统主要依赖本地存储，但如果后续需要增加数据库支持、导出导入或云同步能力，只需在数据层进行扩展，而不必重写表示层和大部分领域逻辑。

最后，编辑器子系统内部对笔迹控制、历史管理、渲染和会话状态进行了进一步拆分，这对于数字笔记本应用尤为重要。因为书写与绘图场景对响应速度和代码结构的要求都较高，模块化拆分能够显著降低编辑器页面本身的复杂度。

### 1.6 隐含设计假设

除了架构图中直接展示的内容外，系统还包含一些隐含设计假设。

第一，系统当前采用本地优先模式，不依赖远程服务器，因此整体设计围绕本地持久化展开。  
第二，系统默认以单用户使用场景为主，并未引入账号体系或多人协作逻辑。  
第三，笔记本、页面与笔迹数据在结构上被明确分离，其中笔迹数据以页面为粒度保存，这样可以降低单次读写的复杂度。  
第四，编辑器在交互过程中优先进行内存中的状态更新，而在一次书写操作或编辑操作完成后再进行持久化，以兼顾交互流畅性和数据安全性。

这些设计虽然未必在图中全部显式标出，但它们对于理解系统当前架构选择具有重要意义。

## 二、UI Design

### 2.1 设计目标

数字笔记本应用的 UI 设计不仅要追求视觉上的整洁，还需要服务于核心使用场景，即“快速进入内容、方便管理笔记、降低书写与整理成本”。因此，本项目在 UI 设计上主要围绕以下几个目标展开：

- 让用户能够快速定位和打开所需笔记；
- 让笔记本管理与分类操作更加直观；
- 让主工作流尽量集中在少量高频入口上；
- 充分利用平板设备的大屏空间，提升信息组织效率。

本报告配套提供了一张主界面 UI 设计图，用于展示系统主要界面的布局与信息结构。

### 2.2 主界面设计说明

从当前 UI 设计图可以看出，系统主界面采用了典型的平板工作台布局，整体由左侧导航区、顶部操作区和中部内容区三部分构成。这种布局形式适合数字笔记类产品，因为它能够在一个页面内同时承载内容浏览、分类切换和快速操作等多种功能。

### 2.3 界面布局分析

#### （1）左侧导航区

左侧导航栏用于承载全局分类与模块切换功能，当前包含：

- 全部笔记
- 最近打开
- 收藏
- 文件夹
- 标签
- 回收站

这一设计让用户能够在固定区域内完成内容筛选与导航切换，符合笔记应用“高频浏览、高频检索”的使用方式。导航栏底部还加入了云空间信息，这虽然不一定是当前已完整实现的功能，但从产品设计角度来看，它有助于强化系统作为完整笔记工具的整体感。

#### （2）顶部操作区

顶部区域集中放置了品牌标识、搜索栏、新建按钮、导入按钮、排序方式、视图切换和用户状态信息。  
这一设计体现出比较明确的主次层级：

- 搜索栏位于顶部显著位置，说明搜索是高频入口；
- “新建”与“导入”按钮强调了内容创建与进入工作流的重要性；
- 排序与视图切换用于提升内容组织效率；
- 用户状态区域则补充了产品完整性和个性化信息。

对于平板设备而言，这种顶部集中式操作布局可以减少用户寻找功能入口的时间，提高整体可用性。

#### （3）中部内容区

中部区域是整个界面的主体部分，主要用于展示最近使用的笔记、文件夹分类以及全部笔记列表。  
其中：

- “最近使用”采用封面卡片布局，便于用户快速回到高频内容；
- “我的文件夹”采用文件夹卡片布局，强化内容的层级组织；
- “全部笔记”则采用列表形式，便于同时展示标题、标签、时间、页数等更多元数据。

这种“卡片 + 列表”的混合设计比较适合笔记类系统。卡片适合突出视觉识别和封面感，列表适合提高信息密度和浏览效率，两者结合能够兼顾美观与实用。

### 2.4 视觉风格与交互特征

从视觉上看，该界面采用浅色背景、大量圆角容器和柔和配色，整体风格较为清爽，符合学习和办公场景下长时间使用的需求。不同卡片和文件夹使用了适度区分的色彩，可以帮助用户更快地识别不同内容区域，同时不会造成过强的视觉干扰。

从交互角度看，该设计具有以下特点：

- 高频操作入口集中，便于快速使用；
- 左侧导航固定，整体可预测性较强；
- 内容区层次清晰，便于用户理解当前所处位置；
- 支持后续自然扩展更多筛选、分类或同步能力。

这些特点说明，该 UI 设计不仅关注静态页面美观，也考虑了日常使用中的实际效率。

### 2.5 UI 设计与系统功能的对应关系

当前 UI 设计图主要展示的是系统主界面或笔记总览界面。虽然它不是编辑页，但它仍然与本项目的核心功能具有较强对应关系。

首先，笔记本管理需要一个统一的内容入口，而该主界面正好承担了“浏览全部内容、查看最近使用、按分类进入目标笔记”的作用。  
其次，搜索、新建和导入等入口体现了系统在“快速创建”和“快速访问”方面的设计重点。  
再次，文件夹、标签、收藏和回收站等导航项说明该 UI 方案具备较好的扩展空间，适合作为数字笔记本应用的主工作台设计。

从完整系统流程来看，用户可以先通过主界面进入目标笔记，再进一步进入笔记本列表页或编辑页，完成页面管理与手写记录等核心任务。因此，这张 UI 图能够较好地体现系统外层工作流的设计思路。

### 2.6 当前 UI 设计的局限与补充说明

需要说明的是，当前这张 UI 设计图更偏向“主界面/总览页”设计，因此它主要体现的是内容组织、导航层级和主入口设计，而不是编辑器中的手写与工具切换细节。

如果希望 Part I 的 UI Design 部分更加完整，后续还可以补充一张编辑器界面的设计图，例如展示页面缩略图侧栏、工具栏、画布区和操作按钮的布局。这样可以更直接地对应项目中已经实现的页面编辑与手写绘图功能。

不过，即使仅从当前主界面设计来看，这张 UI 图也已经较好地体现了系统在信息组织、操作入口和产品整体感方面的设计思路，因此仍然适合作为报告中的主要 UI 设计成果之一。

## 三、总结

总体而言，本项目在架构上采用了较为清晰的分层设计，并在表示层使用 MVVM 风格组织页面与状态逻辑，使系统能够较好地支撑数字笔记本应用中的笔记本管理、页面组织和手写绘图等核心功能。通过 Repository 模式与本地持久化机制，系统实现了结构明确、职责清晰且便于扩展的本地优先架构。

在 UI 设计方面，当前主界面设计图展示了较强的信息组织能力和产品完成度，能够较好地体现数字笔记本应用在大屏设备上的布局特点和主要交互思路。结合架构图与 UI 设计图，可以较为完整地说明本项目在系统结构与界面设计两方面的总体方案。
